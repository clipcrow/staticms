# Daily Report Format

本プロジェクトでは、日々の進捗共有と戦略的改善サイクルの維持のために、以下のフォーマットを使用した日報（Daily
Standup Report）を作成することを義務付けます。

## 目的

1. **コンテキストの連続性**:
   翌日の開発者が前日の思考プロセスと戦略的決定を即座に把握し、迷いなく作業を開始できるようにする。
2. **戦略的修正 (Pivots)**:
   単なる「やったこと」の羅列ではなく、「戦略的マイルストーン」や「アーキテクチャ上の気づき」を記録することで、プロジェクトの質を持続的に向上させる。
3. **リスク管理 (Stop Issues)**:
   ブロッカーや解決すべき課題を明確にし、翌日の優先順位を決定する。

## 書き方フォーマット

以下はテンプレートです。各セクションの趣旨に従って記述してください。

```markdown
# Daily Report - DAY {XX}

**日付**: YYYY-MM-DD **フェーズ**: {現在の開発フェーズ (例: Phase 4: Strategy
Implementation)}

## 作業報告

**趣旨**: 今日行った作業の「戦略的な意味」と「具体的な成果」を記述します。
単にタスクをリストアップするだけでなく、「なぜそれが重要だったか」「プロジェクト全体にとってどのようなマイルストーンとなったか」を強調してください。

**記述項目**:

- **戦略的マイルストーン**:
  v1からの移行、基盤構築、主要機能のアンロックなど、プロジェクトの進路に影響を与えた大きな達成事項。
- **完了したタスク**: 具体的なUS（User Story）、テスト、実装内容。
- **テスト状況**: ユニットテストの通過状況。リントエラー

## 気づきと改善点

**趣旨**:
作業を通じて得られた技術的な知見、テストのコツ、設計上の注意点などを記録します。
これは「未来の自分」や「他の開発者」への貴重なナレッジベースとなります。

**記述項目**:

- **技術的なハマりポイントと解決策**:
  テストライブラリの癖、フレームワークの仕様など。
- **アーキテクチャ上の発見**:
  リントルール、命名規則、ディレクトリ構造に関する改善案。

## STOP ISSUE

**趣旨**:
現在進行中のブロッカー（開発を停止させる問題）や、翌日に持ち越した重大なバグ・課題を記述します。
ここになにもなければ「なし。すべてのテストはGreen。」と記述します。

**記述項目**:

- テスト失敗の原因
- 未解決のバグ
- 仕様の矛盾や不明点
```
