# Daily Standup Report Format (YESTERDAY-TODAY-STOP ISSUE)

本プロジェクトでは、日々の進捗共有と戦略的改善サイクルの維持のために、以下の「YESTERDAY-TODAY-STOP
ISSUE」フォーマットを使用した日報（Daily Standup
Report）を作成することを義務付けます。

これらのレポートは `architecture/daily_standup/` 配下に `DAY{XX}.md`
という命名で保存されます。

## 目的

1. **コンテキストの連続性**:
   翌日の開発者が前日の思考プロセスと戦略的決定を即座に把握し、迷いなく作業を開始できるようにする。
2. **戦略的修正 (Pivots)**:
   単なる「やったこと」の羅列ではなく、「戦略的マイルストーン」や「アーキテクチャ上の気づき」を記録することで、プロジェクトの質を持続的に向上させる。
3. **リスク管理 (Stop Issues)**:
   ブロッカーや解決すべき課題を明確にし、翌日の優先順位を決定する。

## 書き方フォーマット

以下はテンプレートです。各セクションの趣旨に従って記述してください。

````markdown
# Daily Standup - DAY {XX}

**日付**: YYYY-MM-DD **フェーズ**: {現在の開発フェーズ (例: Phase 4: Strategy
Implementation)}

## YESTERDAY

**趣旨**: 前日に行った作業の「戦略的な意味」と「具体的な成果」を記述します。
単にタスクをリストアップするだけでなく、「なぜそれが重要だったか」「プロジェクト全体にとってどのようなマイルストーンとなったか」を強調してください。

**記述項目**:

- **🌟 戦略的マイルストーン**:
  v1からの移行、基盤構築、主要機能のアンロックなど、プロジェクトの進路に影響を与えた大きな達成事項。
- **完了したタスク**: 具体的なUS（User Story）、テスト、実装内容。
- **テスト状況**: E2Eテスト、ユニットテストの通過状況。

## 💡 気づきと改善点

**趣旨**:
作業を通じて得られた技術的な知見、テストのコツ、設計上の注意点などを記録します。
これは「未来の自分」や「他の開発者」への貴重なナレッジベースとなります。

**記述項目**:

- **技術的なハマりポイントと解決策**:
  テストライブラリの癖、フレームワークの仕様など。
- **アーキテクチャ上の発見**:
  リントルール、命名規則、ディレクトリ構造に関する改善案。

## STOP ISSUE

**趣旨**:
現在進行中のブロッカー（開発を停止させる問題）や、翌日に持ち越した重大なバグ・課題を記述します。
ここになにもなければ「なし。すべてのテストはGreen。」と記述します。

**記述項目**:

- テスト失敗の原因
- 未解決のバグ
- 仕様の矛盾や不明点

## TODAY

**趣旨**: 前日の成果と課題（STOP
ISSUE）を踏まえ、当日に達成すべき具体的な目標を定義します。
ここでの記述が、翌日の朝のプロンプトの基礎となります。

**記述項目**:

- 着手するUser Story（US）
- 具体的なTDDステップ（Red -> Green の計画）

---

## 🤖 明日用のプロンプト

作業終了時に、翌日の作業開始時に使用するプロンプトをここに記述します。
これには「戦略的ウォームアップ」の手順を含め、開発者が正しいマインドセットで開始できるようにします。

**必須事項**:

1. **ドキュメントの網羅的指定**: `architecture/`
   配下のドキュメントを**全て個別にリストアップし**、それらを読み込むよう指示を含めてください。「フォルダごと」ではなく、具体的なファイル名を列挙することで、AIがコンテキストを確実にロードできるようにします。

2. **重要ファイルの明示**:
   その日の作業に特に関係する、**特に重要なドキュメントやソースコード**について、別途明示して重点的に読み込ませるセクションを設けてください。

   例:
   ```markdown
   **Work Context**:

   - `architecture/v2/USER_STORIES.md` (特に US-03, US-04)
   - `src/server/api/config.ts` (実装対象)
   ```

3. **重要: コミュニケーションとコミットのルール (厳守)**:
   以下の指示を必ずプロンプトに含めてください。
   - **言語**: 全てのコミュニケーションを**日本語**で行う。
   - **コミットとプッシュ**:
     - 機能実装や修正完了後、必ずユーザーに確認を依頼し、**ユーザーの承認を得てから**コミットする。
     - **コミットメッセージは必ず日本語**で記述する。
     - コミット後は必ず **`git push` を実行**する。
````

## 運用ルール

1. **毎日の終了時**: 作業終了時に必ず作成する。
2. **言語**:
   チーム全体の共有言語（現在は日本語）で記述する。見出しのみ英語（YESTERDAY,
   STOP ISSUE, TODAY）とする。
3. **更新**:
   必要に応じてこのフォーマット自体を改善し、プロジェクトの進化に合わせる。
